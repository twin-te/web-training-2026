**この演習で一番大事なステップです。**
はじめにに書いた通り VM は時間課金なので、消し忘れがこの演習における最大の事故です。必ず以下を実行してください。

1. Cloud Manager でインスタンス(`training-<あなたのGitHubユーザー名>`)の右のメニューから **Delete** を選び、確認ダイアログに従って削除する
2. **Firewalls** から `training-<あなたのGitHubユーザー名>-fw` も削除する(Firewall 自体は無料ですが、掃除まで含めて演習です)
3. [発展の公開](/infra/publish/)をやった人は、**DNS レコードの削除**を運営に依頼する(理由は下記)
4. Linodes の一覧から自分のインスタンスが**消えている**ことを確認する

DNS レコードを消す理由は、お金ではなく**セキュリティ**です。VM を削除すると、あなたが使っていた IP アドレスはクラウド事業者に返却されて、いずれ**他の誰か**に割り当てられます。DNS レコードを残したままだと、`<あなた>.training.twinte.net` がその他人のサーバを指すことになります(ぶら下がったままの DNS、**dangling DNS** と呼びます)。twinte.net の名前で他人のコンテンツが配信できてしまう、立派な乗っ取りの入り口です。「作ったものと逆順に消す」は片付けの鉄則です。

そして最後に、**隣の人とペアになって、お互いの画面で一覧から消えていることを相互チェック**してください。「自分は消したつもり」が一番危ないので、他人の目で確認するところまでが儀式です。

チェックが終わったらインフラ編の演習は完了です。おつかれさまでした!
あなたは今日、自分の PC で動いていたアプリをサーバに引っ越し、ファイアウォールで守り、本番と同じ方式で動かし、そして責任を持って畳みました。立派なインフラ入門者です。

## 次にやってみたいこと(発展)

時間が余った人・もっとやりたい人向けの種です。順不同、どれも運営や先輩に相談しながらどうぞ。

- 削除する前に `/var/log/auth.log` を眺めて、世界の広さ(と怖さ)を体感する
- 本番の [`update_course.sh`](https://github.com/twin-te/twin-te/blob/master/infra/staging/app/cronjob/update_course.sh) のように、講義データの更新を cron で**毎日自動実行**するようにしてみる
- 本番のように **app と db を2台の VM に分けて**、VPC(プライベートネットワーク)でつなぐ(Linode 作成時に VPC を設定できます。db 側のファイアウォールで 5432 を「VPC の内側からだけ」許可するのがポイントです)
- Twin:te 本体の開発に興味が湧いたら、モノレポ [twin-te/twin-te](https://github.com/twin-te/twin-te) のコードをのぞいてみる(front も back も、今日動かしたあの子たちの中身です)

<details><summary>運営メモ(参加者は読まなくてOK)</summary>

- **アカウント**: 参加者はチームの Akamai アカウントに**制限付きユーザー**として追加する。請求情報や他人・本番インスタンスの操作権限を渡さないこと。招待は演習前日までに済ませ、当日はログイン確認から始められるようにする
- **コスト**: Linode 4GB (`g6-standard-2`) は月額 $24($0.036/時)の時間課金で、1人あたり演習時間(数時間)で数十円程度。全員分でも数百円オーダー。ただし消し忘れると1台あたり月数千円になるので、演習翌日に `training-` プレフィックスのインスタンスが残っていないか一覧を確認する(命名規則はこの検出のためにある)
- **ghcr.io**: `ghcr.io/twin-te/twin-te` のイメージは public なので、pull に認証は不要
- **KdB への負荷**: `docker compose run parser` は各自が KdB から直接ダウンロードする。参加者が多い場合は一斉実行にならないよう声をかけるか、誰かが取得した `data/kdb.json` を `scp` で配り回して parser のステップを飛ばす運用も検討する
- **共有 env**: Google OAuth の共有 env の配布手段を決めて、[1. 自分の PC で動かす](/infra/pc/)の TODO 箇所に記入する。VM への持ち込みは `scp` かエディタ貼り付けを案内
- **発展(公開)の対応**: 希望者ごとに、Cloudflare の twinte.net ゾーンへの A レコード追加(DNS only)と、共有 OAuth クライアントへのコールバック URL 追加(1人1件、上限100件)が必要。サブドメインの命名規則を決めて [発展ページ](/infra/publish/)の TODO に記入する。**演習後は DNS レコードとコールバック URL の削除まで確認**(dangling DNS 防止)
- **TA 配置**: SSH 周り(特に Windows: OpenSSH の有無、鍵のパーミッション、PowerShell と WSL の混在)で詰まる人が必ず出るので、SSH に強い TA を1人張り付ける想定でいるとよい
