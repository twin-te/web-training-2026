勉強会の環境構築などで**すでにやったことがある人**は、動くことをもう一度確認して、後半の「何が動いているのか」から読んでください。

READMEの手順は、大きくこの流れです。

1. リポジトリを clone する
2. Docker イメージを**ビルド**する(`docker compose build ...`)
3. 環境変数(`back/.env.local`)を設定する ← **共有 env が必要。下記参照**
4. DB のマイグレーションをする
5. KdB から講義情報を取得して DB に入れる
6. `docker compose --profile docker up` で立ち上げる

このページの残りは、README に書けない情報(共有 env の受け取り方)と、先輩たちが実際に踏んできた詰まりどころ集です。README を進めながら、詰まったらここに戻ってきてください。

## 事前に必要なもの

- **Git と GitHub アカウント**: リポジトリの clone に使います。
- **Docker Desktop(Windows) / OrbStack(macOS)**: コンテナをまとめて動かすためのツールです。
  - Windows の人は [Windows 向け環境構築ガイド](/windows-setup/)に沿って Docker Desktop を WSL ごと入れるのがおすすめです
  - macOS / Linux の人は[バックエンドコースの Phase 1「Docker の導入」](/backend/2-practice/phase1/)に案内があります(macOS は OrbStack を使います)
  - 「Docker って何?」という人も同じく Phase 1 に説明があります。今日のところは「PC の中に小さなサーバたちをまとめて立ててくれる道具」と思っておけば大丈夫です

## 共有 env(環境変数)の受け取り方

README の手順3で、`back/.env.local` に Google OAuth の設定値(`OAUTH_GOOGLE_CLIENT_ID` / `OAUTH_GOOGLE_CLIENT_SECRET`)を書く必要があります。これは Google ログインを動かすための秘密の値で、**当然 GitHub には書いてありません**。

## よくある詰まりどころ Q&A

**Q. `docker compose up` したら「port is already allocated」と怒られた**

A. 必要なポート(4000, 8080, 5432 など)を別のプロセスが使っています。ありがちなのは、以前に起動した別プロジェクトのコンテナ(バックエンドコースの掲示板の PostgreSQL も 5432 を使います!)や、ローカルに直接入れた PostgreSQL です。`docker ps` で動いているコンテナを確認して `docker stop <名前>` するか、犯人が分からなければ周りの人と一緒に `lsof -i :5432` などで探してみましょう。

**Q. イメージのビルドが途中で失敗する・異様に遅い**

A. まず疑うのはメモリ不足です。Twin:te はフロント・バック・DB などを全部ビルドするので、それなりにメモリを食います。Docker Desktop の人は Settings → Resources でメモリ割り当てを増やして(8GB 程度あると安心)、再実行してみてください(OrbStack はデフォルトでメモリを自動調整してくれますが、必要なら Settings → System から上限を変更できます)。ディスク容量不足の場合は `docker system prune` で使っていないイメージを掃除するのも効きます。

**Q. 起動はしたけど、アクセスするとエラーが出る(relation ... does not exist など)**

A. DB のマイグレーション(README の手順4)を飛ばしていませんか?マイグレーションは「DB にテーブルの枠組みを作る」作業で、これをやらないと back がテーブルを見つけられずにエラーになります。README のマイグレーションのコマンド(`migrate-up`)を実行してから、立ち上げ直してください。コマンドが2行(DB_URL と TEST_DB_URL)あるのは、開発用とテスト用の2つの DB があるからです。両方実行しましょう。

**Q. 動いてはいるけど、授業検索しても何も出てこない**

A. KdB からの講義情報の取り込み(README の手順5)を飛ばしていませんか?テーブルは空っぽのまま作られるので、parser でデータを取ってきて投入するまで検索結果はゼロ件です。エラーではないのでかえって気づきにくい罠です。

**Q. Google ログインができない・ログイン後に変な画面に飛ぶ**

A. `back/.env.local` の OAuth の値が正しく設定されているか確認してください(コピペ時に前後の空白や引用符が紛れ込むのもよくあるミスです)。また、アクセスは必ず `http://localhost:4000` からにしてください。別の URL(IP アドレス指定など)からだと、認証後の戻り先が合わずにログインが壊れます。

ここに載っていない詰まり方をしたら、それはあなたが新しい知見を発見したということです。周りのTAに聞いて解決し、よければこのページに追記する PR を出してください。資料もまた、みんなでメンテナンスするプロダクトです。

## 動作を検証してみよう

ブラウザで http://localhost:4000 を開いて、Twin:te の画面が表示されたら成功です!Google ログインをして、授業を検索して、自分の時間割に追加してみてください。いつも使っているあの Twin:te が、いまあなたの PC の中だけで動いています。

## 何が動いているのか

起動した状態で、モノレポのディレクトリで次を実行してみてください。

```sh
$ docker compose ps
```

いくつものコンテナが並んでいるはずです。主要な登場人物はこの4人です。

- **proxy** (nginx) — `localhost:4000` で全部のリクエストを受け付ける玄関。URL のパスを見て、`/api` や `/auth` なら back に、それ以外は front に振り分けます(リバースプロキシ)
- **front** (Vue) — ブラウザに表示される画面
- **back** (Go) — API サーバ。ログインや時間割の処理をする
- **db** (PostgreSQL) — ユーザーや授業のデータの置き場所

「玄関(proxy)が front と back に振り分けて、back が db と話す」——この絵は、このあとサーバに引っ越しても、そして本番環境でも、まったく同じです。

## でも、これはあなたの PC の中だけの話

いま動いている Twin:te は、あなたの PC の中に**しか**ありません。PC のふたを閉じれば止まりますし、隣の人がアクセスすることもできません。

本物の Twin:te がそうであるように、**24時間動き続けて、インターネットのどこからでも使える**ようにするには、どこか別の場所——サーバ——で動かす必要があります。
